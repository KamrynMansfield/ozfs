Open-Zoning Feed Specification!

Introduction

The Open-Zoning Feed Specification (OZFS) is an attempt to create a more universal structure for digitally storing municipality zoning requirements. It is meant to be a digital replica of what is written in the municipality zoning codes and not an interpretation of said zoning codes. This universal, computer-readable format can be used to analyze current zoning standards across multiple municipalities.

Efforts to create the OZFS were started by Luke Reeve and Paul Salama under the direction of Carole Voulgaris. Kamryn Mansfield and Houpu Li built off of this work to finalize the OZFS.

OZFS consists of one GeoJSON file for each municipality. The GeoJSON file contains district geometries and zoning requirements for the municipality. Some benefits of using a GeoJSON file are:

  • Organization: GeoJSON files allow many types of data, including spatial data, to be stored in the same file. Having one file for each municipality may keep things simple.

  • Consistency: GeoJSON files don’t need to include fields with no value. Because some zoning codes may need fields that other zoning codes don’t need, a GeoJSON file will eliminate the need to have a bunch of blank fields taking up space in the file This will allow consistency in naming and organization without including unneeded fields.

  • API Compatibility: GeoJSON files are compatible with APIs. If, one day, we want to create an API with this data, it will be easy to query.

The following sections on this page will describe how the GeoJSON file is structured according to OZFS standards. The naming convention page defines each possible field in the GeoJSON file and shows how it should be written in the code. This slide presentation may also be useful in visualizing the file structure.

Municipality File Structure

A typical GeoJSON file contains a collection of features. Each feature has a list of properties and a geometry. Different from typical GeoJSON files, OZFS records global municipality information under a metadata field. The images below illustrate this structure.

Metadata Fields and Example

Features

In the OZFS municipality file, each feature represents a zoning district with zoning regulations recorded in the feature properties and the district’s boundary recorded in the feature geometry.

Properties

Because zoning regulations can be pretty extensive, the feature properties are organized into four categories. Under each category there are items that help describe information found in the zoning code. The items either describe details or constraints of the district. You can click on the categories below to view the items each one contains.

District Information Items

The items under the district information category have to do with basic information about the district. Besides the permitted uses item, each item has one value connected to it. The permitted uses item is a list containing the following items.

Value An array containing the permitted uses in the district
Exceptions Allowed A true/false value stating whether any exceptions are permitted for the land uses permitted
Exception A description of when the exception is allowed chosen from a list of exceptions described by OZFS
Exception Value An array containing the permitted uses after an exception is granted.

Constraint Details

The items under the constraints categories have to do with regulations found in the city code that constrain aspects of the lot, structures, and other things within the district. Due to the differences among municipalities in the way the zoning laws are written, each constraint item is another list of details that help accomplish three main objectives.

  1. Describe the constraint

  2. Describe exceptions to the constraint

  3. Record complex or conditional constraints

The list below shows all the possible details that can be placed under a constraint item.

Unit The unit of measurement the recorded value will have
Uses An array of uses that share the constraint value
Minimum Value A list containing the value or any formulas needed to obtain the minimum constraint value
Maximum Value A list containing the value or any formulas needed to obtain the maximum constraint value
Exceptions Allowed A true/false value stating whether any exceptions are permitted for the specified land use and constraint
Exception A description of when the exception is allowed chosen from a list of exceptions described by OZFS
Exception Mimimum Value A list containing the value or any formulas needed to obtain the minimum constraint value when the exception is granted
Exception Maximum Value A list containing the value or any formulas needed to obtain the maximum constraint value when the exception is granted

Constraint Values

As depicted in the image above, the constraint details will contain at least one constraint value field. There are four possible constraint value fields: for the minimum and maximum values of that constraint along with the minimum and maximum values of the constraint if exceptions are granted. To accommodate for zoning codes that don’t give a fixed value for their constraints, the constraint value fields are nested list with many other fields that can be used if necessary. The possible fields under the constraint values are listed below. Note that these can be a little easier to understand when visualizing it. See examples for extra clarity.

Expression / Expressions Expression is a value or equation telling how to calculate the value. Expressions (plural) is an array with multiple expressions. This is only used inside a rule where you must choose the between more than one expression (see “Select Type” below).
Rules An array containing different expressions that will be chosen if certain conditions are met.
Logical Operator In case there are two conditions that have to be met (ex. greater than 50 feet AND less than 100 feet), the logical operator field specifies the relationship that the conditions must have. “AND” or “OR”
Conditions

An array of conditions that have to be met in order for the expression to be chosen to assign the value. These conditions are the “if”s in an if-then statement.

Symbol The symbol used to set the condition (“>”, “<=”, etc.)
If-Expression 1 and If-Expression 2

These are the expressions being compared that come before and after the symbol.

Example:

Condition: “When the building is between 20 and 40 feet tall, the front setback is 50 feet.”

If Expression 1 Symbol If Expression 2
“bldg_height” “>=” “20”
“bldg_height” “<=” “40”
Value The value connected to the condition. (ex. “variable” “symbol” “value” = “height” “>” “20”)
Select Type For cases when one must choose the largest or smallest of a set of values.
NOTE: Expression Creation Warning

When an expression that is more than just a value is needed, be careful to write it correctly. Any variables must be written according to the OZFS naming convention.
