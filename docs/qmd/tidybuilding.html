The tidybuilding Data Class

Introduction

This page is to document the process we took in creating a tidybuilding data class.

The goal was to, in the simplest way possible, describe a building with enough detail to confidently show the space it takes up in a parcel and say whether it meets zoning requirements or not. Here are some thoughts we had on the needed info for the tidybuilding class.

In order to properly represent a building, these are some things we may need:

  • Building Type: This could be stated somewhere as part of the object, or we could assume the type based on the number of units and bedrooms.

  • Building Height: Total height could be stated somewhere, but it would probably be more beneficial to have the height for each floor. I just wonder what we would do for parts of the building that add to the height but donâ€™t have a floor plan (like the roof).

  • Building Footprint: Just one floor may suffice in the simplest of occasions, but for occasions where the setback changes as the building getâ€™s taller, we may need to have a footprint for each floor as well.

  • Floor Area: Different from the footprint, we may need a separate polygon to calculate floor area. Or we could just have a field that states the floor area.

  • Bedrooms: Sometimes the number of bedrooms affects the zoning constraint (ex. one parking stall / bedroom) or certain areas have a bedroom limit, so it might be nice to record this.

  • Unit Amount: For similar reasons as stated with bedrooms. Sometimes there are restrictions on the number of units or the density of the units.

  • Roof Pitch: This could probably be a global variable as well.

  • Parking Plans and Location: Even if a building would fit in a parcel, the parking requirements may take up more room and essentially change the buildable area. It would be important to know if parking is part of the building plan and where it will be.

    • Remember that parking doesnâ€™t usually need to have a setback, so the buildable area for the building would be a bit different from the builddable area for the parking.
    • Some codes also have requirements for parking location. (ex. enclosed two-car parking attached to house required, garage doors must not be street facing, etc.)

Getting Prototype Buildings

Before coming up with a concrete data structure to store building information, we wanted to look at some building examples to get an idea of what info one would need to look for when determining if it will fit in the parcel. Here is where we got the building prototypes:

  • Boston 3D Data Download: This database stores 3D models of all of Boston.
  • Boston Property Assesment: This data set is where we could get the number of units to help us decide which buildings to select for the prototypes.
